# Опис тестового покриття

## Методика тестування

У межах розробки програмного застосунку було створено окремий тестовий проєкт `MyBlazorApp.Tests`, призначений для автоматизованої перевірки коректності роботи основної бізнес-логіки. Для реалізації тестів використано фреймворк `xUnit`, який дозволяє виконувати як перевірку окремих фіксованих сценаріїв, так і параметризоване тестування на основі наборів вхідних даних.

Під час побудови тестового набору були використані два основні підходи:

1. Атрибут `[Fact]` застосовано для перевірки окремих чітко визначених сценаріїв, де результат має бути однозначним. До таких випадків належать, наприклад, успішний вхід користувача в систему, створення початкових файлів даних, зміна статусу прийому, видалення запису або вихід користувача із системи.
2. Атрибут `[Theory]` використано для параметризованих тестів, коли одна і та сама логіка перевіряється на різних наборах вхідних даних. Такий підхід було застосовано для перевірки правил валідації email, пароля, обов’язкових полів, вартості різних послуг, обробки помилкових комбінацій логіна і пароля, а також для аналізу кількох статусів запису.

Застосування двох типів тестів дозволило охопити не лише позитивні сценарії роботи системи, але й граничні та помилкові випадки, що підвищує достовірність перевірки.

Тестування охоплює такі основні модулі системи:

1. `DataService`.
Перевірено створення файлового сховища, ініціалізацію початкових даних, додавання нових записів, оновлення інформації про прийом, зміну статусу, видалення записів, перемикання стану оплати, пошук користувачів за email, сортування списків та коректне визначення вартості послуг.

2. `AuthService`.
Перевірено успішні та неуспішні сценарії входу, зміну авторизаційного стану, очищення поточного користувача при виході, спрацьовування події `OnChange`, реєстрацію нового користувача та обробку спроби повторної реєстрації з уже наявним email.

3. Моделі `User` та `Appointment`.
Окремо протестовано правила валідації, реалізовані через `DataAnnotations`. Перевірено обов’язковість ключових полів, правильність формату email, мінімальну довжину пароля, наявність вибраної послуги, лікаря, телефону, часу та інших критичних параметрів.

Загалом у тестовому проєкті реалізовано 80 юніт-тестів, що забезпечують перевірку різних аспектів функціонування застосунку.

## Результати тестування

Після завершення написання тестів було виконано автоматичний запуск тестового набору за допомогою команди:

```bash
dotnet test Term_Paper.sln
```

У процесі виконання було здійснено повну перевірку тестового проєкту разом з основним застосунком. Усі тести завершилися успішно, що свідчить про коректність реалізованої логіки та стабільність основних програмних модулів.

Підсумкові результати тестування:

1. Загальна кількість тестів: 80.
2. Успішно пройдено: 80.
3. Кількість помилок: 0.
4. Кількість пропущених тестів: 0.

Отримані результати підтверджують, що реалізовані сервіси та моделі працюють відповідно до очікуваних сценаріїв. Наявність повного набору юніт-тестів дозволяє знизити ризик виникнення регресійних помилок у майбутньому, спрощує супровід програмного забезпечення та підвищує загальну надійність системи.
